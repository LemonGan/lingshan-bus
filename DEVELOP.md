# 开发规范 - gastack 工作流程

## 评审流程
```
Think → Plan → Build → Review → Test → Ship → Reflect
```

## 每阶段任务

| 阶段 | 任务 | 输出 |
|------|------|------|
| **Think** | 分析需求，明确要解决什么问题 | 问题定义 |
| **Plan** | 设计方案，考虑边界情况和数据结构 | 方案设计 |
| **Build** | 按方案实现代码 | 功能代码 |
| **Review** | 用 `/review` 标准严格评审 | 问题列表 |
| **Test** | 实际测试，验证功能正常 | 测试结果 |
| **Ship** | 部署/发布 | 上线确认 |
| **Reflect** | 总结经验教训 | 改进建议 |

## 代码评审标准

### Blocking（阻塞上线）
- 用户无法使用的基本功能问题
- 导致崩溃的bug
- 安全漏洞

### Required（必须修复）
- 逻辑错误
- 未处理边界情况
- 性能问题

### Suggestions（建议优化）
- 代码重复
- 可维护性问题
- 缺少注释

## 常用命令

- `/review` - 代码评审
- `/qa` - 测试验证
- `/ship` - 发布
- `/document-release` - 更新文档
